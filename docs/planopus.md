# Аудит проекта AKLAB — рекомендации и план работ

> Дата аудита: 06.07.2026 (v1.1.13). Аудит проведён по исходному коду
> (api/, app/, services/, lib/, scripts/, docs/, infra/, CI/CD).
> Это план улучшений, а не описание багов — баги и history уже
> зафиксированы в `docs/compact-doc.md` (gotchas #1-52).

## Как читать этот документ

Каждый раздел — область аудита. Внутри — находки (что есть) и
рекомендации (что делать). В конце — сводный чеклист с приоритетами
(P0 — критично/безопасность, P1 — важно, P2 — желательно).

## 1. Архитектура (общая оценка: 7/10)

### Что хорошо
- Чёткое разделение: Strapi (API+БД) / Vue (frontend) / микросервисы-парсеры
  (по одному на источник) / `@aklab/sqlite-queue` как шина сообщений.
  Это правильная архитектура для проекта такого размера — не over-engineered
  (нет Kafka/RabbitMQ там, где хватает SQLite-очереди), не under-engineered
  (парсеры изолированы, могут падать независимо).
- `services/_shared` (`@aklab/service-shared`) — общий код (config, logger,
  health-server, queue-worker, strapi-client, anti-ban, city-detect,
  property-classifier) переиспользуется 10 сервисами. DRY соблюдается
  на уровне библиотек.
- Pipeline Orchestrator (`api/src/services/pipeline.ts`) с персистентным
  состоянием в БД + SSE — грамотное решение для реалтайм-прогресса без
  polling и с поддержкой reconnect при перезагрузке страницы.
- `docs/compact-doc.md` как единый "context file" для онбординга новой
  сессии/разработчика — отличная практика, редко встречается в небольших
  проектах.

### Проблемы

**P1. Монолитная точка отказа — общий `queue.db` (SQLite, WAL).**
Все 10+ сервисов пишут в один файл БД. При росте нагрузки (текущий polling
200ms) это станет узким местом. Также единая точка отказа: если файл
повреждён/заблокирован — весь pipeline встаёт. Для текущего масштаба
(десятки-сотни задач/час) это приемлемо, но нужно заложить план миграции
на Redis/BullMQ при росте (см. `.clinerules` — Redis уже в стеке проекта
для других задач, логично применить и здесь).

**P1. api/ и app/ не входят в npm workspaces.**
`package.json` root: `"workspaces": ["lib/*", "services/*"]` — но `api/`
и `app/` собираются и устанавливаются отдельно (`cd api && npm install`,
`cd app && npm install`). Это причина части проблем с деплоем
(несовпадение версий зависимостей, необходимость трёх отдельных
`npm install` в `deploy-prod.sh`). Рекомендация: включить `api` и `app`
в workspaces (или явно задокументировать, почему это невозможно — Strapi
и Vite не всегда дружат с hoisting зависимостей, но это решается через
`nohoist`/pnpm).

**P1. Несогласованность окружения Node между PM2 daemon и процессами.**
Задокументировано в gotchas #23 (better-sqlite3 NODE_MODULE_VERSION
mismatch) — происходило минимум дважды. Нет `.nvmrc` в корне репозитория,
нет `engines` в package.json. Рекомендация: добавить `.nvmrc` (зафиксировать
Node 22.x), добавить `engines.node` во все package.json, чтобы `npm install`
падал явно при несовпадении версии, а не тихо ломался при рантайме.

**P2. Deploy-скрипт как единственный источник orchestration (405 строк bash).**
`scripts/deploy-prod.sh` вручную перечисляет список сервисов в 3 разных
местах (build loop, pm2 restart list, health-check loop) — уже приводило
к багу (gotcha #21: новый сервис не попал в список → MODULE_NOT_FOUND).
Рекомендация: вынести список сервисов в один JSON/YAML-манифест
(`services/services.json` — slug, port, health_port), генерировать
bash-массивы и `ecosystem.config.js` из него программно. Также стоит
рассмотреть переход на Docker Compose + Traefik (уже используется для
инфраструктуры, см. `infra/traefik/`) вместо PM2 — единообразие деплоя,
меньше ручной синхронизации версий Node.

**P2. Деплой пишет обратно в git (version bump + commit + push с прод-сервера).**
`deploy-prod.sh` делает `git commit`/`git push origin main` из под
production-процесса. Это анти-паттерн GitOps: source of truth должен
формироваться в CI, а не на проде. Частично решено флагом `--ci`
(CI делает bump и коммит сама), но non-CI режим всё ещё пишет в репо
с прод-сервера. Рекомендация: полностью перевести релизы на CI-режим,
manual-режим оставить только для экстренного hotfix без версионирования.

## 2. Безопасность (общая оценка: 6/10)

### Что хорошо
- `.env` в `.gitignore`, секреты не закоммичены (проверено `git ls-files`).
- `safeEval` в `focusEngine.ts` — осознанный отказ от `eval()`/`new Function`
  напрямую на пользовательском вводе: whitelist переменных, regex-фильтр
  опасных символов перед вычислением. Это лучше, чем ничего, хотя не
  идеально (см. ниже).
- JWT-аутентификация на всех endpoints, публичная роль ограничена
  login/register.
- `scripts/check-env.js` — pre-deploy проверка на placeholder-значения
  секретов (`change_me_`, `tobemodified`) — хорошая практика.

### Проблемы

**P0. `safeEval` в `api/src/services/focusEngine.ts` всё ещё использует
`new Function(...)`.** Хотя вход фильтруется regex'ом (`[a-zA-Z_$]` и
`[;{}[\]\\\`]`), это compile-time риск: `condition_value` правил берётся
из БД (`focus-rule` content-type), которая редактируется через админку.
Если когда-то появится публичный API для правил (или XSS/CSRF позволит
внедрить правило через админскую сессию) — это RCE. Рекомендация: заменить
на настоящий safe-expression evaluator без `Function`/`eval`
(например, простой recursive-descent parser на 4 операции +/−/×/÷ и
сравнения, ~80 строк кода, полностью detereministic). Пример есть в
множестве npm-пакетов (`expr-eval`, `mathjs` в safe-mode) — либо написать
вручную, т.к. грамматика тривиальна.

**P0. Raw SQL с интерполяцией строк в `focusEngine.ts` (batch update).**
```js
const scoreCases = updates.map(u => `WHEN ${u.id} THEN ${u.score}`).join(' ');
const tagCases = updates.map(u => `WHEN ${u.id} THEN '${JSON.stringify(u.tags).replace(/'/g, "''")}'`).join(' ');
await s.db.connection.raw(`UPDATE properties SET focus_score = CASE id ${scoreCases} ...`);
```
`u.score` вычисляется из `safeEval` (потенциально контролируемое значение,
если правило скомпрометировано) и подставляется в SQL без параметризации.
Ручное экранирование кавычек (`replace(/'/g, "''")`) — хрупкий паттерн,
типичный источник SQL injection в исторических CVE. Рекомендация:
использовать параметризованные запросы (`?` placeholders + массив
параметров) или полностью заменить raw SQL на batch update через
`entityService`/`db.query().updateMany()` в цикле — производительность
пострадает незначительно (сотни записей, не миллионы), а риск исчезнет.

**P1. `getFocus` в `property.ts` строит SQL с конкатенацией условий.**
Сами значения параметризованы (`?` + массив `params`), но `sortField`
жёстко берётся из whitelist-таблицы `allowedSorts` — это правильно
сделано. Единственный риск — если в будущем кто-то добавит новое поле
сортировки без валидации. Рекомендация: зафиксировать паттерн явным
комментарием "ВСЕГДА добавлять новые поля в allowedSorts, никогда не
интерполировать `query.sort` напрямую" — чтобы не сломали при рефакторинге.

**P1. Управление секретами: рассинхронизация `.env` ↔ PM2 daemon env
(gotcha #24, #33 в deploy-prod.sh).** `API_TOKEN_SALT` и другие секреты
существуют в двух источниках правды одновременно (файл `.env` и
environment PM2-процесса, захваченный в момент `pm2 start`). Deploy-скрипт
содержит целый блок кода (`Step 0.5`) для авто-синхронизации через grep
+ node one-liner. Это хрупко и уже приводило минимум к двум инцидентам
(401 на всех сервисах). Рекомендация: перейти на явную загрузку `.env`
через `dotenv` в самом entry-point каждого процесса (а не полагаться на
PM2 env caching), либо использовать `pm2 restart --update-env` в каждом
деплое как обязательный шаг вместо ручной синхронизации через grep/sed.

**P1. `JWT_SECRET` отсутствие в `.env` роняет все сессии при рестарте
(gotcha #32).** Уже случалось. Рекомендация: `scripts/check-env.js`
уже проверяет обязательные переменные — убедиться, что `JWT_SECRET`
входит в `requiredVars` (сейчас в списке есть только `APP_KEYS`,
`JWT_SECRET`, `ADMIN_JWT_SECRET`, `API_TOKEN_SALT`, `TRANSFER_TOKEN_SALT`,
`STRAPI_API_TOKEN`, `DATABASE_FILENAME` — **проверить**, что это
единственный источник валидации и он реально блокирует деплой, а не
просто предупреждает).

**P2. CORS/Traefik/Node-RED/Redis — рекомендации по стеку из
`.clinerules`.** Текущий проект не использует Redis и Node-RED, хотя
инфраструктурные best-practices пользователя ориентированы на этот стек.
Для AKLAB конкретно: если нагрузка вырастет, Redis pub/sub мог бы заменить
самодельный SSE-broadcast в `pipeline-sse.ts` (сейчас работает только
в рамках одного Node-процесса — если когда-нибудь Strapi будет
масштабирован на несколько инстансов за Traefik load balancer, SSE-клиенты
на разных инстансах не получат broadcast от других). Пока Strapi — один
процесс, это не проблема, но заложить в план роста.

**P2. Rate limiting упомянут в E2E-тестах ("5 skipped из-за rate limiter"),
но не задокументирован явно.** Стоит явно задокументировать конфигурацию
rate limiter в `api/config/middlewares.ts` (если он там) — сейчас это
всплывает как побочный эффект в тестах, а не как осознанная защита.

## 3. Backend (Strapi) — оценка 7/10

### Что хорошо
- Чёткая структура content-types (Property, Setting singleton,
  MarketReference, UserComment, CronLog, Source, FocusRule, Pipeline).
- `focusEngine.ts` — понятный, документированный код с чёткими
  JSDoc-комментариями, разделение responsibilities (scoreProperty vs
  scorePropertiesBatch).
- Использование batching (200 записей за раз) в `scorePropertiesBatch`
  для больших датасетов — правильный паттерн, не грузит всё в память.
- `buildParseRules()` — DRY helper, переиспользуется и в API, и в
  парсерах (хотя дублируется как код в двух местах — см. ниже).

### Проблемы

**P1. Дублирование `buildParseRules` между `api/` и `services/_shared`.**
Задокументировано самим проектом (gotcha #48) как осознанный компромисс
("api/ не зависит от @aklab/service-shared"). Рекомендация: пересмотреть
это решение — добавить `api/` как консьюмера `@aklab/service-shared`
(это чисто TS-пакет без Strapi-специфичных зависимостей) технически
возможно и устранит дублирование логики фильтрации (стоп-слова, цена,
площадь, города), которая сейчас может разойтись при правке одного из
двух файлов и не другого.

**P1. Отсутствие unit-тестов у парсер-сервисов (кроме `_shared`,
`analyzer`, `digest`).** Судя по структуре `services/parser-*/` — нет
директорий `__tests__/`. При этом каждый парсер содержит нетривиальную
логику извлечения данных (регэкспы для цены/площади, детекция города,
DOM-селекторы). Это именно тот код, который чаще всего ломается при
изменении верстки сайта-источника (что уже происходило многократно —
gotchas #7-9, #26-28 в compact-doc). Рекомендация: добавить unit-тесты
хотя бы для чистых функций парсинга (extractPrice, extractArea,
detectCity per-parser overrides) — используя фикстуры реального HTML,
сохранённые в `__fixtures__/`. Это не требует Playwright — тестируется
`cheerio.load(fixture)` + функция извлечения.

**P1. `api/src/api/property/controllers/property.ts` смешивает уровни
абстракции.** `getFocus` содержит SQL-запросы прямо в контроллере
(244 строки, из них ~120 — построение SQL). По конвенциям Strapi
бизнес-логика должна жить в `services/`, контроллер — тонкий слой.
Рекомендация: вынести SQL-построение в `services/property.ts` (или
переиспользовать `focusEngine`-подобный сервис), контроллер должен
только парсить query params и вызывать сервис.

**P2. `pipeline.ts` — 640 строк в одном файле.** Оркестрирует parseAll,
analyze, digest, cancel, state management. Для читаемости стоит разбить
на модули: `pipeline/state.ts` (get/update state), `pipeline/stages.ts`
(parseAll/analyze/digest), `pipeline/index.ts` (orchestration/run/cancel).
Сейчас функционально работает, но при дальнейшем росте (например, добавление
новых стадий) 640-строчный файл станет неудобен для навигации.

**P2. `cron/controllers/cron.ts` — 265 строк, вероятно тоже содержит
устаревшую логику после перехода на Pipeline Orchestrator.** Стоит
проверить, не остались ли там дублирующие pipeline старые
analyze/digest-эндпоинты, которые больше не используются UI (compact-doc
упоминает, что cron теперь делегирует в `pipeline.run()` — если
контроллер оставляет старые прямые вызовы parallel-путём, это источник
рассинхронизации логики).

## 4. Парсер-микросервисы — оценка 6/10

### Что хорошо
- Общий шаблон (`_shared`) даёт единообразие: health-server, queue-worker,
  strapi-client, anti-ban (случайный UA, задержки, retry с backoff),
  city-detect, property-classifier — всё вынесено централизованно.
- 3-фазный парсинг (parse → existence check → fetchDetails) с корректным
  прогрессом (needed фиксировано, fetched растёт) — качественная инженерия
  UX для долгих операций.
- `parse-handler.ts` — общий обработчик очереди для всех парсеров, dry.

### Проблемы

**P1. `package.json` парсер-сервисов рассогласованы.** Сравнение
`parser-fabrikant/package.json` vs `parser-alfalot/package.json`
показало: разные версии Playwright (`^1.52.0` vs `^1.60.0`), у alfalot
отсутствуют зависимости `better-sqlite3`, `cheerio`, `express`, `winston`,
`zod`, `@types/better-sqlite3`, `@types/express`, отсутствует `start`
script, нет финального newline. Похоже, что `alfalot` — более старая
версия шаблона сервиса, не обновлённая при рефакторинге остальных.
Рекомендация: зафиксировать единый шаблон `package.json` (например,
Yeoman-generator или простой bash-скрипт `scripts/new-parser.sh`,
который копирует эталонный шаблон и заменяет только slug/port),
привести все 10 сервисов к единой версии зависимостей. Уже есть
`docs/adding-source.md` — стоит проверить, соответствует ли инструкция
актуальному состоянию, и добавить в неё автоматизированный скрипт
генерации вместо ручного копирования.

**P1. `queue-worker.ts` дублируется практически 1-в-1 между сервисами**
(diff показал разницу только в строковых литералах — имени очереди
и лог-сообщении). Рекомендация: вынести общий `createQueueWorker(queueName,
handler)` в `_shared`, каждый сервис вызывает его с одной строкой
конфигурации вместо копии файла.

**P2. Наименования конфигурации портов разбросаны в 3+ местах**: `compact-doc.md`
(таблица портов), `ecosystem.config.js`/`ecosystem-local.config.js`,
`scripts/deploy-prod.sh` (health-check список), возможно и в seed-данных
Source. Единый источник правды (см. рекомендацию из раздела 1 про
services.json-манифест) устранил бы риск рассинхронизации (уже
происходившей — "следующий свободный порт: 1348" в документации
означает, что нумерация ведётся вручную и может конфликтовать).

**P2. Anti-ban модуль (`_shared/src/anti-ban.ts`, 82 строки) —
рандомный UA + задержки + retry.** Это разумный базовый набор, но
некоторые источники (`fedresurs`) блокируются на уровне TLS fingerprint
(Qrator), что anti-ban модуль принципиально не решает. Стоит явно
задокументировать границы возможностей anti-ban (что решает — soft
rate-limiting/UA-detection, что не решает — TLS fingerprinting/WAF) —
чтобы не тратить время на попытки "починить" fedresurs текущими
средствами (уже отражено в docs, но стоит вынести в сам код как
комментарий в anti-ban.ts).

## 5. Frontend (Vue 3 + Vite) — оценка 6.5/10

### Что хорошо
- Composition API + `<script setup>` — соответствует best practices
  Vue 3 (и требованиям `.clinerules`).
- Pinia для auth state, TypeScript везде.
- Composables (`useFocusTab`, `usePropertyData`, `useTheme`) — попытка
  выносить переиспользуемую логику из компонентов, `usePropertyData.ts`
  (87 строк) — хороший пример: чистая, тестируемая единица с явным
  API (properties/focusProperties/loading/error).
- Skip-to-content ссылка, `aria-live`, `aria-label` везде в `App.vue` —
  внимание к accessibility, что нечасто встречается в внутренних
  админ-панелях.
- Тёмная тема через CSS-переменные (`--accent`, `--text-muted`, ...) —
  чистая реализация без дублирования цветов.

### Проблемы

**P1. `PropertyListView.vue` — 1118 строк.** Это самый большой файл в
проекте и явный кандидат на рефакторинг. Содержит, судя по структуре
приложения, минимум 3 таба (Все объекты/В фокусе/В работе), фильтры,
пагинацию, collapsible панель запуска pipeline. Рекомендация: разбить
на подкомпоненты по табам (`PropertyAllTab.vue`, `PropertyFocusTab.vue`,
`PropertyInProgressTab.vue`) + вынести фильтры в отдельный
`PropertyFilters.vue` + композабл для пагинации (`usePagination.ts`).
Целевой размер — не более 200-300 строк на компонент.

**P1. `DocumentationView.vue` — 850 строк статического контента.**
Если это преимущественно markup с описанием архитектуры — стоит вынести
контент в markdown-файлы (`docs/*.md`) и рендерить через markdown-to-html
на этапе сборки, а не хардкодить HTML в Vue-компоненте. Это упростит
поддержку документации (не нужно трогать Vue-код чтобы поправить текст)
и убережёт от рассинхронизации между `docs/compact-doc.md` (источник
правды для AI-сессий) и `DocumentationView.vue` (источник правды для
пользователей UI) — сейчас это два независимых места с одной и той же
информацией об архитектуре.

**P1. `SettingsView.vue` — 584 строки, 4 таба (Дайджест/Правила/Парсеры/Эталоны).**
Уже частично разбит на панели (`RulesPanel`, `ParsingRulesPanel`,
`SourcesPanel`, `MarketReferencesPanel` — хорошо), но сам `SettingsView.vue`

запуска pipeline, которая уже есть в `PropertyListView.vue` (согласно
compact-doc — "3 триггера в UI" для одного и того же pipeline). Вынести
общую логику запуска/статуса pipeline в composable `usePipeline.ts`

оба view.

**P2. Только 1 файл в `app/src/composables/__tests__/`** (`useTheme.test.ts`).

чистые, легко тестируемые функции (мокается `axios`-инстанс). Рекомендация:
добавить тесты хотя бы для happy-path + error-path каждого composable —
это дешёвые тесты (без DOM), быстро окупятся при рефакторинге.

**P2. UI-библиотека не используется.** `.clinerules` рекомендует
PrimeVue/Quasar для Vue-проектов, но в AKLAB весь UI — кастомные
Tailwind-компоненты. Для админ-панели с таблицами/фильтрами/пагинацией
готовая библиотека компонентов (DataTable, MultiSelect) сократила бы
объём кода в `PropertyListView.vue`/`PropertyTable.vue` в разы. Это
не критично (проект работает), но стоит рассмотреть при следующем
крупном рефакторинге фронтенда.

## 6. Тестирование — оценка 6.5/10

### Что хорошо
- 250 unit-тестов (vitest), 29 E2E (Playwright) — весомое покрытие
  для проекта такого размера.
- Тесты для критичной бизнес-логики: `buildParseRules` (14 тестов),
  `createProperty` (rules enforcement), очередь, pipeline, cron, focusEngine.
- `docs/e2e-test-inventory.md` и `docs/unit-test-inventory.md` — явный
  учёт того, что покрыто, а что нет — отличная практика прозрачности.

### Проблемы

**P1. Парсер-сервисы (10 штук) не имеют собственных unit-тестов**
(см. раздел 4). Это самый нестабильный код в проекте (парсинг живых
сайтов), и одновременно наименее покрытый тестами.

**P1. 5 skipped E2E-тестов + нерешённая проблема network isolation
для прод-тестов** (задокументировано в compact-doc "Известные баги").
E2E-тесты на прод нельзя запускать с dev-сервера. Рекомендация:
настроить отдельный GitHub Actions job, запускаемый из сети,
имеющей доступ к прод-домену (использовать self-hosted runner на
прод-сервере, либо публичный runner — раз домен публичный
`aklab.tirobots.ru`, GitHub-hosted runner должен иметь доступ через
интернет, а не только internal network; стоит перепроверить, почему
именно timeout — возможно, firewall/pfSense блокирует внешние IP GitHub
Actions).

**P2. Нет нагрузочного/стресс-тестирования очереди.** SQLite-очередь
с polling 200ms — при увеличении числа одновременных задач (несколько
источников с depth=5000) неизвестно поведение под нагрузкой. Рекомендация:
простой скрипт нагрузочного теста (`scripts/load-test-queue.js`) —
накидать 10k задач, замерить throughput/latency, зафиксировать baseline
для будущего сравнения.

## 7. CI/CD и деплой — оценка 6/10

### Что хорошо
- `deploy-prod.sh` — заметная зрелость: backup БД перед деплоем,
  health-check с retry, rollback при ошибке (`trap rollback ERR`),
  синхронизация версий Node, AI-генерация changelog, Telegram/email
  уведомления. Это далеко не "скопировал файлы и перезапустил".
- CI: `ci.yml` (тесты на PR), `deploy-dev.yml` (ручной), `deploy-prod.yml`
  (авто при push в main + ручной с confirm) — разумное разделение окружений.
- `[skip ci]` в release-коммитах предотвращает рекурсию.

### Проблемы

**P1. Нет branch protection ни на `main`, ни на `dev`** (явно
задокументировано в compact-doc). Это значит: force-push в main возможен,
прямой push в main без PR технически не блокируется на уровне GitHub
(хотя процесс требует PR "по конвенции"). Рекомендация: включить branch
protection на `main` (минимум: запретить force-push, требовать
прохождения `ci.yml` перед merge). Для `dev` — как минимум запретить
force-push, чтобы не терять историю (уже был инцидент — "странный коммит
795fdbf" в истории main).

**P1. `trap rollback ERR` хрупкий** (задокументировано в известных
багах: "может не сработать при exit 1 из health check"). Рекомендация:
явно тестировать rollback-путь (например, в staging — намеренно сломать
build и убедиться, что rollback срабатывает end-to-end), не полагаться
только на happy-path проверку.

**P2. Auto-deploy на прод при каждом push в main** — агрессивная
конфигурация для проекта без branch protection и без обязательного
code review gate. Рекомендация: либо добавить обязательный review
(хотя бы self-review чеклист), либо держать auto-deploy только для
`dev`, а `main → prod` — ручное подтверждение (частично уже есть —
"ручной запуск с confirm", но и "авто-деплой при push" одновременно
существует, что создаёт двусмысленность: кто и когда решает, что мерж
в main = немедленный прод-деплой).

**P2. Версия бампается вручную в `deploy-prod.sh` (`parts[2]++`)
без учёта semver-семантики** (всегда patch, никогда minor/major).
Для проекта на этой стадии это нормально, но если появятся breaking
changes в API — стоит предусмотреть механизм ручного указания
major/minor bump (например, по commit message convention
`feat!:`/`BREAKING CHANGE:`).

## 8. Наблюдаемость (observability) — оценка 4/10

**P1. Отсутствует централизованный сбор логов и алертинг об ошибках.**
Логи — только `pm2 logs` на каждом сервере отдельно. При 15 процессах
на 2 серверах (prod+dev) поиск причины инцидента требует ручного
`ssh` + `pm2 logs --nostream` на нужный процесс. Рекомендация (в духе
стека из `.clinerules` — Redis уже рекомендован для проекта):
- Минимальный вариант: агрегация логов через `pm2-logrotate` +
  единый лог-файл с ротацией, отправка критичных ошибок в Telegram
  (частично уже есть — notify на деплой, но не на рантайм-ошибки).
  Sentry/Bugsnag были бы идеальны для error tracking, но это внешний
  сервис — если проект принципиально self-hosted, можно рассмотреть
  self-hosted GlitchTip (Sentry-совместимый) на существующей
  Docker/Traefik инфраструктуре.
- Метрики: `pm2 monit`/`pm2 web` дают базовые метрики, но нет
  исторических графиков (CPU/RAM/queue depth во времени). Рассмотреть
  простую интеграцию Prometheus + `pm2-metrics` exporter, раз Docker
  уже используется для Traefik.

**P2. Health-check есть (все 12 сервисов + Strapi + App), но нет
alerting на "сервис лежит X минут"** — только ручная проверка при деплое.
Рекомендация: простой cron-скрипт (`scripts/health-check.js` уже
существует!) — расширить его до периодического запуска через
`node-cron`/system cron с отправкой Telegram-алерта при недоступности
> 5 минут, а не только при деплое.

## 9. Документация — оценка 8/10 (лучшая часть проекта)

### Что хорошо
- `docs/compact-doc.md` — образцовый "living document": архитектура,
  gotchas (52 пронумерованных пункта!), session handoff notes, известные
  баги. Редчайшая практика для проектов такого размера — обычно такие
  знания живут только в голове разработчика.
- `docs/insights/` — отдельная директория для инсайтов по датам, что
  предотвращает разрастание `compact-doc.md` до бесконечности (хотя
  сам файл уже 787 строк — см. ниже).
- Множество `docs/plan*.md` — история планирования по фичам, что
  позволяет проследить "почему" за архитектурными решениями.

### Проблемы

**P2. `compact-doc.md` (787 строк) растёт линейно с каждой сессией**
и начинает содержать одновременно (а) актуальную архитектуру, (б) историю
изменений по датам, (в) TODO/известные баги, (г) gotchas. Рекомендация:
разделить на:
- `docs/architecture.md` — только текущее состояние (без истории).
- `docs/gotchas.md` — только пронумерованные gotchas (стабильный
  reference, редко меняется).
- `docs/sessions/YYYY-MM-DD.md` — session handoff notes (то, что сейчас
  в конце compact-doc.md, "Сделано в сессии...") — по одному файлу на
  сессию, как уже частично сделано для `docs/insights/`.

Это не блокирует работу, но текущий формат ("прочитай 787 строк перед
началом работы") создаёт trade-off между полнотой контекста и скоростью
онбординга новой AI-сессии/разработчика.

**P2. `DocumentationView.vue` (850 строк, для пользователей) и
`docs/compact-doc.md` (787 строк, для AI-сессий) описывают одну и ту
же архитектуру в двух независимых местах** — уже отмечено в разделе 5.
Синхронизация вручную = риск расхождения информации со временем.
