# AKLAB — план улучшения UI/UX (красивее · удобнее · современнее)

> Дата: 06.07.2026 (v1.1.26). Аудит фронтенда `app/src` (Vue 3 + Vite +
> Tailwind, кастомная дизайн-система на CSS-переменных).
> Цель — сделать интерфейс визуально современнее, удобнее в работе и
> проще в поддержке, не ломая существующую тёмную/светлую тему и a11y.

## Как читать

Сначала — честная оценка текущего состояния (что уже хорошо). Затем —
находки и предложения, сгруппированные по темам, с приоритетами
(P1 — заметный эффект/быстро, P2 — важно, P3 — nice to have). В конце —
поэтапная дорожная карта и чеклист.

## 0. Текущее состояние: что уже хорошо

Фронтенд не «с нуля» — база зрелая, ломать её не нужно:

- **Дизайн-токены** в `tailwind.css`: цвета, тени, радиусы, переходы,
  glass-эффект — через CSS-переменные. Тёмная (дефолт) и светлая темы.
- **Доступность (a11y):** skip-to-content, `aria-label`, `aria-live`,
  `:focus-visible` ринг, semantic `<nav>`/`<main>`.
- **Микро-взаимодействия:** page-transition (fade), hover-scale карточек,
  skeleton-загрузчики, `.card-interactive`.
- **Адаптивность:** мобильное меню-гамбургер, responsive-таблица,
  переключатель «таблица/карточки» (`ViewToggle`).
- **Компонентная декомпозиция:** табы вынесены в под-компоненты,
  форматтеры и часть логики — в `utils/`/`composables/`.

Итого: фундамент 7/10. Проблемы — не в отсутствии дизайна, а в его
**несистемности**: стили копируются инлайном, цвета хардкодятся, нет
переиспользуемых UI-примитивов. Это бьёт по консистентности и скорости
доработок. Ниже — как это исправить и где добавить «вау».

## 1. Дизайн-система: убрать инлайн-стили, ввести семантику (P1)

**Проблема: тотальный `style="..."` с CSS-переменными.**
Практически каждый элемент во всех view написан так:
```html
<h1 style="color: var(--text-main)">…</h1>
<div style="background: var(--bg-elevated); border-color: var(--border-subtle)">
<span style="background: rgba(79,140,255,0.15); color: #4f8cff">Торги</span>
```
При этом в `tailwind.css` уже есть утилиты (`.text-content`,
`.bg-surface-elevated`, `.border-theme`, `.glass`), но они почти не
используются. Инлайн-стили: (а) многословны, (б) нельзя переиспользовать,
(в) не подчиняются hover/focus/dark вариантам Tailwind, (г) провоцируют
рассинхронизацию (один и тот же «бейдж торгов» описан по-разному в
`PropertyCard` и `DetailView`).

**Рекомендация:**
1. Расширить набор утилит-классов и перейти на них:
   `.text-content` / `.text-muted` / `.bg-elevated` / `.border-theme` /
   `.card` (радиус + фон + бордер + тень одним классом).
2. Для Tailwind 4 — завести токены как `@theme` переменные, чтобы работали
   классы `bg-elevated`, `text-muted`, `border-subtle` нативно (без inline).
3. Правило на будущее: **инлайн-`style` только для действительно
   динамических значений** (например, цвет по score), всё статичное — класс.

**Проблема: нет семантических цветовых токенов.**
Статусы/скор/отклонение раскрашены хардкодом hex по всему коду:
`#ef4444`, `#f59e0b`, `#10b981`, `#4f8cff`, `rgba(251,191,36,0.15)` —
в `Dashboard`, `PropertyCard`, `PropertyTable`, `PropertyDetailView`,
`useFocusTab`. Токенов `--success/--warning/--danger/--info` нет (есть
только `--error-bg/--success-bg`).

**Рекомендация:** ввести семантические токены и шкалу score:
```css
--success: #10b981; --success-soft: rgba(16,185,129,.15);
--warning: #f59e0b; --warning-soft: rgba(245,158,11,.15);
--danger:  #ef4444; --danger-soft:  rgba(239,68,68,.15);
--info:    #4f8cff; --info-soft:    var(--accent-soft);
--score-hot: #ef4444; --score-warm: #f59e0b; --score-cold: #4f8cff;
```
и одну функцию/утилиту `scoreColor()`/`statusStyle()` — единый источник
(сейчас `scoreColor` дублируется в `DashboardView` и в `useFocusTab`).

## 2. Библиотека UI-примитивов (P1)

**Проблема: кнопки, бейджи, карточки, инпуты пишутся заново в каждом файле.**
Например, «primary-кнопка» встречается десятками копий:
```html
<button class="px-4 py-2 rounded-xl text-sm font-semibold text-white …"
        style="background: var(--accent)">…</button>
```
Аналогично — бейджи-пилюли, метрик-тайлы, карточки-контейнеры,
текстовые инпуты. Это главный источник визуальной несогласованности
(разные `rounded-lg`/`rounded-xl`, разные паддинги, разные оттенки).

**Рекомендация:** собрать тонкий слой базовых компонентов в
`app/src/components/ui/`:
- `BaseButton.vue` — варианты `primary | secondary | ghost | danger`,
  размеры `sm | md`, состояние `loading`/`disabled`, слот иконки.
- `BaseBadge.vue` — `variant` (success/warning/danger/info/neutral),
  единый радиус-pill; заменит все разрозненные бейджи (статус, торги,
  отклонение, теги).
- `BaseCard.vue` — контейнер (`bg-elevated` + `border-theme` + радиус +
  тень), опционально `interactive` (hover-lift).
- `BaseInput.vue` / `BaseSelect.vue` — единый вид полей форм.
- `StatCard.vue` — KPI-карточка дашборда (заголовок, число, иконка,
  дельта, клик → маршрут).
- `EmptyState.vue` — иконка + текст + CTA для пустых списков.
- `Tag.vue` — тег фокуса (сейчас стилизуется через `tagStyle`).

Эффект: `DashboardView`/`PropertyCard`/`DetailView` сократятся, а вид
станет единым. Это дешевле и легче, чем тянуть весь PrimeVue/Quasar
(хотя для сложных виджетов — см. §8).

## 3. Иконки: единый набор вместо emoji + SVG вперемешку (P2)

Сейчас в UI смешаны emoji (`🔥 📊 📸 📋 ⚠️ ↻ ✕ ‹ ›`) и инлайновые SVG
(тема, гамбургер, CIAN-пин). Emoji рендерятся по-разному на разных ОС,
не подчиняются `currentColor`, выглядят «несерьёзно» для аналитического
инструмента.

**Рекомендация:** подключить `lucide-vue-next` (лёгкий, tree-shakeable,
stroke-иконки в стиле текущих SVG). Завести `<Icon name="flame" />`
обёртку. Заменить emoji на иконки, наследующие `currentColor` и размер.
Останется единый визуальный язык, темизируемый вместе с темой.

## 4. Дашборд: из «двух цифр» в информативную панель (P1)

**Проблема:** дашборд беден. Всего 2 KPI (Всего / В фокусе), список
горячих и пилюли по типам. Нет динамики, нет «что нового», нет статуса
парсинга. Для мониторингового продукта это ключевой экран — он должен
отвечать «что изменилось и на что смотреть» за 3 секунды.

**Рекомендации:**
- **Ряд KPI-карточек (4-5)** через `StatCard`: Всего · В фокусе ·
  🔥 Горячие (score≥50) · Недооценённые · Новые за 24ч. С иконкой и,
  где есть данные, дельтой («+12 за сутки»).
- **Мини-график** динамики (новые объекты/в фокусе по дням, 14 дней) —
  спарклайн или столбики. Данные — из `property-event`/`createdAt`.
- **«Объекты по типам» → горизонтальный бар-чарт** (доля каждого типа),
  а не просто пилюли — нагляднее распределение.
- **Статус парсеров** мини-виджетом (сколько online, когда последний
  парсинг) — сейчас это спрятано в Настройках.
- **Лента последних событий** (вошёл в фокус, изменилась цена) — есть
  `property-event`, показать 5-10 свежих.

**Важно (перф UX):** `fetchTypeBreakdown` тянет `pageSize: 5000` и
считает типы на клиенте — дашборд грузится долго. Нужен агрегатный
endpoint `GET /properties/stats` (счётчики по типам/статусам/фокусу
одним запросом). Это и UI, и backend-задача — вынести в общий бэклог.

## 6. Детальная страница объекта: иерархия и «герой» (P2)

**Проблема:** `PropertyDetailView` (513 строк) — плоская стена
`label/value` в трёх-колоночной сетке. Цена (главная ценность продукта)
подана тем же кеглем, что «Тип торгов». Взгляд не цепляется за главное.

**Рекомендации:**
- **Hero-блок сверху:** крупно цена + `₽/м²`, рядом бейдж отклонения
  («−34% к рынку») и score-индикатор; заголовок, адрес, ключевые
  метрики (площадь, тип) — в один заметный ряд.
- **Второстепенные поля** (даты, источник, id, эталон) — в сворачиваемую
  секцию «Детали» или в две колонки помельче.
- **Фотогалерея — всегда** (сейчас только для `is_undervalued`), с
  плейсхолдером, если фото нет. Lightbox уже есть — добавить свайп на
  мобильном и клавиши ←/→ (сейчас только кнопки).

- **Sticky action-bar** (сменить статус / комментарий) внизу или сбоку,
  чтобы не скроллить к действиям на длинной карточке.
- Разбить файл на под-компоненты (`DetailHeader`, `DetailFacts`,
  `PhotoGallery`, `EventTimeline`, `CommentBox`) — заодно к §2.

## 7. Обратная связь и состояния (P1)

**Проблема: нет системы уведомлений (toasts).** Успех/ошибка сейчас —
маленький красный текст внизу страницы (`<p style="color:#fca5a5">`),
а успешные действия (сменил статус, добавил комментарий, поставил в
очередь фото) — вообще без подтверждения. Пользователь не уверен,
сработало ли.

**Рекомендации:**
- Ввести **toast-систему** (composable `useToast()` + `ToastHost`
  в `App.vue`): success/error/info, авто-скрытие, стек. Использовать во
  всех мутациях (status change, comment, pipeline start/cancel, CSV).
- **Единые пустые состояния** через `EmptyState` (иконка + текст + CTA)
  вместо «Нет данных» простым текстом (Dashboard, списки, комментарии).
- **Единые состояния ошибки** списков: не только текст, но «Повторить».
- **Оптимистичные обновления** для смены статуса/тегов — мгновенный
  отклик + откат при ошибке (сейчас `saving` блокирует кнопку и ждёт).
- Индикатор **фонового прогресса pipeline** (мини-бейдж в шапке со
  статусом «Идёт парсинг… 42/100»), т.к. процесс долгий и запускается
  с разных экранов — единый глобальный индикатор через `usePipeline`.

## 8. Доступные интерактивные примитивы (P2)

**Проблема:** несколько интерактивных элементов — не с клавиатуры.
- Кликабельные `<div>` на дашборде (KPI-карточки, «горячие объекты»)
  используют `@click` без `role="button"`/`tabindex`/обработчика
  клавиш — недоступны с клавиатуры и скринридеру.
- Клик по строкам таблицы (`<tr @click>`) — тоже не фокусируется.
- Навигация карточек через `router.push` в обработчике вместо
  `<router-link>` — теряется middle-click/Ctrl+click, hover-preview.

**Рекомендации:**
- KPI-карточки и «горячие объекты» → **`<router-link>`** (семантика,
  клавиатура, новые вкладки бесплатно).
- Строки таблицы → добавить `tabindex="0"` + `@keydown.enter`, либо
  обернуть первую ячейку в `<router-link>`.
- Для меню/дропдаунов/диалогов/табов рассмотреть **headless-библиотеку**
  (Reka UI / Radix-Vue или Headless UI) — она даёт доступные, но
  нестилизованные примитивы (focus-trap, ARIA, клавиатура), которые
  красим своими токенами. Это точечное усиление, а не замена дизайна
  на PrimeVue целиком.

## 9. Движение и «дорогое» ощущение (P3)

- **`prefers-reduced-motion`:** нигде не учитывается. Обернуть анимации
  (page-transition, hover-scale, skeleton-pulse, slide-down меню) в
  `@media (prefers-reduced-motion: reduce)` и отключать — требование
  a11y и хороший тон.
- **Переходы страниц** — сейчас только fade. Добавить лёгкий сдвиг/масштаб
  (8-12px, 200мс) для ощущения глубины (с учётом reduced-motion).
- **Стаггер** появления списков/карточек (небольшая задержка по индексу)
  — недорого, заметно «оживляет».
- **Единые тайминги** — использовать `--transition-fast/med` из токенов
  везде (сейчас часть анимаций хардкодит `0.2s`, `0.25s`).

## 10. Типографика и сетка (P3)

- **Шрифт Geist** заявлен в `font-family`, но не видно `@font-face`/
  подключения — вероятно, фолбэк на Inter/system. Либо подключить Geist
  (self-hosted, чтобы без внешних CDN на self-hosted проекте), либо
  честно оставить Inter и убрать неиспользуемое имя.
- **Типо-шкала:** ввести токены размеров (`--text-xs…3xl`) и
  придерживаться их; сейчас кегли расставлены точечно (`text-xl`,
  `text-2xl`, `text-lg`, `text-sm`, `text-xs`).
- **Единый радиус:** в разметке вперемешку `rounded`, `rounded-lg`,
  `rounded-xl`, `rounded-full`, а токены `--radius-*` не задействованы.
  Свести к 2-3 значениям через классы.
- **Ширина контента:** есть `--content-max: 1280px` и
  `max-w-screen-xl` — использовать единообразно (сейчас в части экранов
  `max-w-4xl`, в части — `max-w-screen-xl`).

## 11. Мелочи UX, дающие много (P2)

- **Фильтр-чипы (`FilterChips`)** мелкие (`text-xs px-2 py-1`) — тач-цель
  < 44px. Увеличить, добавить счётчик выбранных и кнопку «Сбросить».
- **Сохранение фильтров/сортировки** в URL query (частично есть для
  `property_type`) — чтобы ссылку на отфильтрованный список можно было
  расшарить и вернуться назад без потери состояния.
- **Массовые действия** в фокусе — вынести в липкую панель над списком
  при наличии выбранных («Выбрано 5: Просмотрено · Отклонить · CSV»).
- **Форматирование чисел** — убедиться в едином разделителе разрядов и
  «₽» (есть `formatPrice`/`formatPriceShort` — довести до всех мест).
- **Скелетоны под контент** — на списках объектов и деталях показывать
  скелетон точной формы (сейчас есть `SkeletonTable`/`SkeletonLoader` —
  применить единообразно на всех экранах загрузки).

- [ ] Фильтры в URL, липкая панель массовых действий, крупные чипы.

## 13. Оценка и ожидаемый эффект

| Направление | Сейчас | Проблема | Эффект после |
|---|---|---|---|
| Дизайн-система | 6/10 | инлайн-стили, хардкод цветов | консистентность, скорость доработок |
| UI-примитивы | 4/10 | всё пишется заново в каждом файле | −30-40% разметки, единый вид |
| Дашборд | 5/10 | беден, медленно грузится | информативность, «что нового» за 3с |
| Детальная | 6/10 | плоская, нет иерархии | акцент на цене/выгоде, карта, фото |
| Обратная связь | 4/10 | нет тостов, тихие успехи/ошибки | пользователь уверен в действии |
| Доступность | 6/10 | div-клики, нет reduced-motion | клавиатура, скринридер, спокойные анимации |
| Иконки/типографика | 5/10 | emoji+SVG, разнобой радиусов | единый визуальный язык |

**Итог:** интерфейс уже опрятный, но «самодельно-несистемный». Главный
рычаг — Фаза 1 (токены + примитивы + тосты): она даёт максимум
консистентности и «современности» при минимуме кода, и на неё
опираются все последующие улучшения экранов. Полный фреймворк
(PrimeVue/Quasar) тянуть не обязательно — достаточно тонкого
in-house слоя компонентов + `lucide` + headless-примитивов для сложных
интерактивов.
