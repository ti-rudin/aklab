# План: Переосмысление парсинга v2

## Текущее состояние (v1.0.53)

### Что работает
- 10 парсеров, 15 PM2 процессов, cron-расписание 03:00–06:00 МСК
- Дедупликация по `(source, external_id)` — unique index
- Generic parse handler — единый для всех парсеров
- Photo-fetcher для недооценённых объектов

### Проблемы

#### P0 — Парсеры не работают
| Парсер | Результат | Причина |
|--------|-----------|---------|
| ГИС Торги | 0 объектов | `isCommercialProperty()` фильтрует всё |
| Сбербанк-АСТ | 0 объектов | Аналогично |
| Росэлторг | 0 объектов | Generic-заглушка, нет пагинации |
| Инвест МО | 0 объектов | Generic-заглушка, нет пагинации |
| Инвест Москва | 0 объектов | Generic-заглушка, нет пагинации |
| Фабрикант | 2 объекта | Коммерческих лотов мало |

#### P1 — Парсинг только со списка
Ни один парсер не заходит на страницу объекта. На входе:
- title (полный)
- цена (из карточки)
- площадь (из title/excerpt, regex)
- URL ссылка

**Чего нет:**
- описание (excerpt — обрезка 500 символов из текста карточки)
- адрес (regex по excerpt, часто пустой)
- координаты (latitude/longitude)
- фотографии (отдельный photo-fetcher, только для недооценённых)
- контактная информация
- дата публикации

Это критично для анализа — без площади и адреса `price_per_sqm = undefined`, объект не считается недооценённым.

#### P2 — Хрупкий external_id
Generic-парсеры генерируют ID из URL: `${slug}-${link.split('/').pop()}`. Если сайт изменит URL-структуру — все объекты получат новые ID → дубли при повторном парсинге.

#### P3 — Generic CSS-селекторы
5 парсеров (m-ets, roseltorg, invest-mosreg, investmoscow + отчасти sberbank-ast) используют универсальные селекторы:
```js
['.card', '.lot-card', '[class*="card"]', 'article', 'tr']
```
Это хватает мусор (навигация, sidebar, footer) или не находит карточки вообще.

#### P4 — 165 из 178 объектов = `city=other`
Парсеры не распознают регион. `detectCity()` ищет слово "москва" в title/excerpt — на торговых площадках адрес редко указан в карточке списка.

## Стратегия

### Принцип 1: Сначала заставить работать существующие парсеры
Не писать новые, а починить то что есть. 6 из 10 парсеров дают 0 объектов — это баг, не фича.

### Принцип 2: Парсинг карточки → парсинг детальной страницы
Переход от "данные из списка" к "данные из карточки объекта". Это:
- описание, адрес, координаты, фото
- точная площадь (не regex из title)
- дата публикации
- контактная информация

### Принцип 3: Site-specific вместо generic
Каждый парсер должен знать HTML-структуру конкретного сайта. Generic-подход не работает — 5 заглушек доказывают это.

## Архитектурные решения

### Двухфазный парсинг vs. парсинг с детальных страниц

**Вариант A: Парсинг карточек (сейчас)**
- ✅ Быстро (1 HTTP-запрос на страницу = 10-50 карточек)
- ✅ Мало нагрузки на сайты
- ❌ Мало данных (title + цена + иногда площадь)
- ❌ Плохой external_id (из URL, не из ID лота)

**Вариант B: Парсинг детальных страниц (Фаза 4)**
- ✅ Полные данные (описание, адрес, фото, координаты)
- ✅ Надёжный external_id (ID лота с детальной страницы)
- ❌ Много запросов (1 на объект = 100-300 запросов за запуск)
- ❌ Долго (5 сек/объект × 200 = ~17 минут)
- ❌ Anti-bot (многие сайты блокируют частые запросы)

**Рекомендация: Гибрид**
1. Список → извлечь URL + базовые данные + external_id → сохранить
2. Для НОВЫХ объектов (не дублей) → открыть детальную страницу → обновить
3. Детальную страницу открываем 1 раз при первом парсинге, потом не трогаем

### isCommercialProperty() — пересмотр

Текущий фильтр отсекает по keywords в title. Проблема: на гос. торгах в title может быть "Нежилое помещение" а в description — "квартира". Или наоборот.

**Решение:**
- Не фильтровать на этапе парсинга
- Фильтровать на этапе анализа (когда есть больше данных с детальной страницы)
- Или: фильтровать по `property_type` (парсер должен классифицировать, а не строковый поиск)

### Generic → site-specific миграция

Каждый парсер должен:
1. Знать URL поиска (с параметрами для коммерческой недвижимости)
2. Знать селекторы карточек (site-specific, не generic)
3. Знать селекторы данных внутри карточки (title, price, area, link)
4. Знать пагинацию (URL-параметр / кнопка "далее")
5. Знать структуру детальной страницы (для Фазы 4)

## Технические заметки

### external_id — надёжные источники
| Источник | Хороший ID | Плохой ID |

| ЕТП РФ | `lot_id` из таблицы | — |
| Агр. банкрот | `lot_id` | — |
| М-ЕТС | — | `link.split('/').pop()` ⚠️ |
| Росэлторг | — | `link.split('/').pop()` ⚠️ |
| Инвест МО | — | `link.split('/').pop()` ⚠️ |
| Инвест Москва | — | `link.split('/').pop()` ⚠️ |
| Сбербанк-АСТ | `purchase_id` | — |

### Rate limiting (anti-bot)
- Между запросами к одному сайту: `2000 + Math.random() * 3000` мс
- Между поисковыми запросами: `3000 + Math.random() * 3000` мс
- Для детальных страниц (Фаза 4): `3000 + Math.random() * 2000` мс
- Retry: 3 попытки с exponential backoff (5с, 15с, 45с)

### Фильтр price_per_sqm
Объекты без цены или площади НЕ сохраняются (createProperty возвращает null). Это правильно — без `price_per_sqm` невозможно определить недооценённость. Но это означает что объекты только с ценой (без площади) теряются. В Фазе 4, когда площадь будет извлекаться из детальной страницы, таких потерь станет меньше.
